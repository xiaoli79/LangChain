### 介绍一下这个项目。

**回答一：（精简版）**

**基于脚手架的微服务在线租房系统**，业务模型对标了贝壳、安居客、闲鱼等流行应用。无论是交互体验、架构设计，还是数据模拟都达到真实的工业水准设计。

本项目是一个 Java 综合项目，涵盖【前端】与【后端】开发，业务上包含房源管理、租户和房东沟通、房源上下架等整套业务流程。核心技术包括微服务、分布式集群、高并发、分布式对象存储、高速缓存、消息队列、即时通信、解决方案设计。

- **后端采用 Spring Cloud 微服务架构**，引入**脚手架**，将系统拆分为**五个微服务**。服务间借助 OpenFeign 实现相互调用。同时，项目集成了 Nginx、Nacos、MySQL 等 10 多个主流 Java 后端组件，技术栈丰富且前沿。
- **前端包含 Web管理端 和 微信小程序端，基于 Vue 3 框架**，使用 Vite 构建工具，以 JavaScript 为主要语言，利用 Axios 高效交互数据，并引入 Element-Plus 组件库，让系统界面既实用又美观。C 端使用了微信小程序作为面向移动端应用入口。

- **数据库**：MySQL + MyBatis
  - MySQL 和 MyBatis 是数据存储和访问的一个常见的做法。MySQL 是一个开源的关系数据库管理系统，它可以免费使用。MyBatis 提供了一种相对简单的方法来执行 SQL 语句，不需要编写复杂的 ORM 映射文件。
- **缓存**：Redis
  - 我需要将热点数据存入缓存中，因为这些内容是用户常常访问的，虽然我只是实现了项目，并没有很多人使用，但我判断如果是真实上线的项目，这部分流量是较大的。为了提高系统性能，我选择了 Redis，它是基于内存的存储系统，其读写速度非常快，可以显著提升数据访问速度。
  - 延申问题：redis的其他特点 (推荐学习redis精品课程)：
    - **高性能**：是基于内存的存储系统，其读写速度非常快，可以显著提升数据访问速度。
    - **丰富的数据结构**：Redis支持多种数据结构，如字符串、哈希、列表、集合、有序集合等，这使得它能够适应各种不同的缓存需求。
    - **持久化支持**：虽然Redis是内存数据库，但它提供了数据持久化的选项，可以将内存中的数据保存到磁盘上，确保数据的安全性和可靠性。
    - **高并发处理能力**：Redis能够处理大量的并发请求，这对于高流量的Web应用来说非常重要。它能够减少对后端数据库的直接访问，从而降低数据库的负担。
    - **分布式缓存**：Redis支持主从复制和哨兵模式，可以构建高可用性的集群架构，提供数据的备份和故障转移功能，保证系统的稳定运行。
    - **解决缓存问题的能力**：Redis能够有效应对缓存穿透、缓存雪崩和缓存击穿等问题，通过设置合理的过期时间、使用布隆过滤器等策略来保证系统的稳定性

    - **路由和消息筛选**：RabbitMQ提供了灵活的路由和消息筛选机制，支持多种交换机类型，可以根据消息的特性将消息路由到不同的队列中。
    - **社区支持和成熟性**：作为一个开源项目，RabbitMQ拥有活跃的社区支持和广泛的用户范围，具有较高的稳定性、可靠性和成熟性。
    - **消息确认和死信队列**：RabbitMQ提供了消息确认机制和死信队列，确保消息在消费过程中的可靠性，以及在消费失败时能够有相应的处理策略。

### 统一异常处理是什么？是怎样进行处理的？

- 统一异常处理：是一种架构设计模式，核心是通过集中化方式管理并响应系统运行中的各类异常。简单讲，接口发生异常时，若不处理，就无法向使用者返回准确提示；但如果每个接口自行处理，既会导致代码冗余、维护困难，还会出现异常响应不统一的问题。因此，统一异常处理旨在解决传统方式的这些痛点，进而提升系统的健壮性、可维护性与用户体验。
- 在比特Java脚手架项目中异常处理分为两部分分别是：（1）网关层异常 （2）业务层异常
  - 对于网关层异常 ：需要在网关服务层实现ErrorWebExceptionHandler接口，当异常发生时，实现ErrorWebExceptionHandler接口的类的handle 方法会自动调用对异常进行统一处理。并返回全局通用的请求响应数据结构。
  - 对于业务层发生的：通过`RestControllerAdvice+ ExceptionHandler`进行处理，返回全局通用的请求响应数据结构。

注意：可参考课件《比特Java脚手架-后端上》统一异常处理部分。



### Java已经有内置的异常类为什么还需要自定义异常？

- 增加代码的可读性同时也符合业务规则和语义：业务逻辑往往有其自身的规则和限制，自定义异常能够更好地体现这些规则。使用自定义异常能够使得代码更加易于理解和维护。
- 满足业务需求：Java提供了一些内置的异常类，但是它们并不一定能够满足我们的具体业务需求，所以我们需要自定义异常类。



### 为什么需要Bean拷贝工具类，项目中为什么要用多种数据模型？

- **为什么需要Bean拷贝工具类：**
  - 在Java web项目中通常将项目分为：控制器（Controller） 层、业务逻辑层（service）层、数据访问（DAO）层。而在不同层之中我们通常会使用不同的数据模型。这也就意味着项目中往往不止一种数据模型，我经常需要将数据从一种数据模型转换为另外一种数据模型，正是因为如此才需要Bean拷贝。
- **为什么要划分不同的数据模型：**
  - **提高代码的可读性和可维护性**：每种对象都有其特定的用途和职责，使得代码结构更加清晰，开发人员能够更容易地理解数据的流动和使用方式。每个对象采取统一的命名，项目会变得更加一致和可预测，从而减少了出错的可能性。
  - **解耦：**DO、BO、DTO、VO等分别作用于不同层，他们的划分降低了各个部分的耦合度。修改某一层的逻辑或数据时减少对于其它层的影响。
  - **优化性能：**不同的对象，比如DTO和VO可以根据自身的功能和当前的需求，进行裁剪和拼装。合理的利用网络传输资源，提高性能。



### 你如何实现的二级缓存？通过二级缓存能够解决什么问题？

在脚手架项目当中我引入了Caffeine作为本地缓存并将本地缓存作为一级缓存，引入Redis作为二级缓存，二者与 MySQL 搭配，组成两级缓存。

**当系统收到查询请求后，先进入缓存查询阶段。**

- 缓存查询阶段：
  - 首先尝试读取一级缓存，若一级缓存存在数据，直接将数据返回给调用方。若一级缓存不存在数据，进入二级缓存查询阶段。
  - 如果二级缓存存在数据先将数据写入一级缓存，再返回数据。若二级缓存也不存在数据：进入数据库查询阶段。
- 数据库查询阶段：
  - 一二级缓存均未命中，被迫查询数据库（DB） 获取原始数据，随后将数据分别写入二级缓存和一级缓存，最后返回数据。

### 为什么文件服务中引入了OSS对象存储文件不使用本地文件存储或其他存储机制如MySQL等？

**引入oss对象存储机制的主要原因如下：**

- 无限扩展性：
  - 本地： 物理容量/性能有限，扩展需加硬件，复杂昂贵。
  - MySQL： 分库分表复杂，存储海量文件效率极低，成本过高。
  - OSS： 架构天然支持近乎无限的水平扩展，按需自动扩容。
- 高可用与持久性：
  - 本地： 单点故障风险高，需复杂备份/容灾方案。
  - MySQL： 文件存储易成单点，备份/恢复海量文件慢。
  - OSS： 数据自动多副本/纠删码跨设备/区域存储，故障自愈，持久性极高。
- 成本效益：
  - 本地： 前期硬件投入大，运维（电力/人力）成本高，资源易浪费。
  - MySQL： 许可/计算资源成本高，存储文件性价比极低。
  - OSS： 按使用量付费，无前期投入；存储单价低；支持冷热分层进一步降本。
- 性能与并发：
  - 本地： 单机I/O易成瓶颈，难应对高并发。
  - MySQL： 文件I/O消耗核心数据库资源，性能差，影响业务。
  - OSS： 分布式架构支持高并发访问；易集成CDN加速全球读取。
- 简化运维：
  - 本地/MySQL： 需专业团队维护硬件/软件/备份/扩容，复杂耗时。
  - OSS： 全托管服务，云商负责底层运维，用户通过API/控制台管理，负担轻。
- 丰富功能：
  - 本地/MySQL： 实现高级功能困难且成本高。
  - OSS： 开箱即用：生命周期管理、版本控制、细粒度权限、加密、图片处理、CDN集成、静态网站等。
- 聚焦核心：
  - OSS 让企业无需自建维护复杂存储设施，专注核心业务开发。

- 回答一（如果前面说项目后端是一个人开发）：据我了解企业中项目都是合作开发，并且采用多分支策略实现高效的并行协作。为了快速适应企业开发模式熟练git操作，所以我采用多分支策略模拟多人协作模式。
- 回答二（如果前面项目回答了后端是多人开发）：采用多分支策略可以实现高效的并行协作，让我们团队的工作推进更加顺利，成员之间工作不互相影响。
- 多分支策略的具体好处：
  - 并行开发与隔离：不同的开发者可使用不同的分支进行开发，避免相互干扰。
  - 环境稳定性保障：不同的环境（开发/测试/生成.......）都对应特定的分支确保环境稳定可靠。
  - 灵活发布控制：我的分支策略中包含bug分支，当线上出现问题时可灵活拆分bug分支修改并及时发布。

**具体分支策略如下：**

- **master 分支**：用于部署到正式发布环境。此分支为唯一分支，并且会被设置为保护分支（不允许任何人推送，只允许仓库管理员合并）。
- **develop 分支**： 含义为开发分支，基于 master 分支创建的唯一分支，并且会被设置为保护分支，始终始终保持最新完成（自测通过）以及 bug 修复后的代码（测试环节发现的bug）。可部署到联调或测试环境。
  - 如果master分支有变更需要及时同步。
- **feature  分支**：通常为新功能或新特性开发分支，以`develop`分支为基础而创建出的分支。
  - 命名以 `feature/` 开头，后面拼接功能名称。例：`feature/map`，多个单词之间可以用短线分割。
  - 新特性或新功能开发完成后，开发人员需合到develop分支，合入develop分支后将其删除。
  - 如果develop分支有变更需要及时同步。
- **bug分支：**修改bug的分支，线上bug基于master分支创建出分支，测试过程发现的bug基于develop分支创建出分支。
  - 命名以`bug/`开头，后面拼接bug名称（能指明是什么bug），多个单词之间可以用短线分割。
  - 如果是线上bug，直接将bug分支部署到测试环境进行测试，测试通过后合入master分支重新发布上线，上线成功后删除bug分支。
  - 非线上bug修复完成后合入develop分支，合入后重新将develop分支代码部署到测试环境，测试人员对所修改bug验证无误后可删除bug分支。

### 讲一下你对字典服务和参数服务的理解？为什么要这么拆分？

字典服务通常用于管理系统中的静态数据，如国家/地区列表、性别、订单状态等。这些数据具有以下特点：

- 变更频率低：如“性别”一般不会频繁修改。
- 全局通用性：多个业务模块可能共用同一套字典数据（如“订单状态”可能被订单、物流等多个服务引用）。
- 数据量较小：通常以KV（Key-Value）结构存储，便于快速查询。

参数服务用于管理动态配置，如系统开关、业务规则阈值、超时时间等，特点包括：

- 变更频率高：例如促销活动的折扣率可能随时调整。
- 业务相关性：不同模块可能有独立参数（如支付超时时间、风控规则）。
- 支持动态生效：通常结合配置中心实现实时更新，避免重启服务。

拆分依据如下：

- 解耦与复用：字典数据通常全局共享，独立服务避免重复存储；参数服务则按业务划分，减少耦合。
- 维护清晰：静态字典和动态配置的变更管理策略不同，拆分后更易于维护。

远程调用性能更高，适合服务间高频调用。HTTP兼容性强，跨系统，跨语言适合前后端分离的场景。



### 你把用户服务模块拆分成B端用户和C端用户分的依据是什么？

把用户服务拆分成B端用户与C端用户的依据如下：

B端用户通常是企业内部员工或者商业用户，这些用户的特点是用户量小，权限高，注册途径少（超级管理员开通）通常是先开通，再登录，对安全要求较高。

C端用户通常是普通消费者用户，这些用户的特点是用户量大，权限第，注册途径多（微信一键登录、短信验证码登录、QQ一键登录）通常是注册即登录尽可能要减少用户的操作次数，降低用户操作成本。

拆分B端用户和C端用户分的本质是业务差异驱动拆分，避免“一刀切”导致系统臃肿或性能瓶颈。



### 短信登录服务，手机验证码为什么要设置有效期？

手机验证码设置有效期是出于安全性、用户体验和系统性能的综合考虑，主要原因包括：

- 防止暴力破解与重放攻击
  - 暴力破解：如果验证码长期有效，攻击者可反复尝试（如遍历6位数字组合），增加破解概率。
  - 重放攻击（Replay Attack）：攻击者截获旧验证码后重复使用（如通过木马窃取短信），有效期限制可降低此类风险。
- 提升用户体验
  - 时效性：用户通常在短时间内完成操作（如登录/注册），过期后需重新获取，避免因旧验证码混淆导致流程卡顿。
  - 减少误操作：例如用户输错手机号，有效期可自动失效，避免无效验证码被滥用。
- 优化系统资源
  - 存储压力：服务端无需长期保存验证码记录，减少Redis等缓存的存储负担。
  - 清理机制：自动过期可简化垃圾数据清理逻辑（如定时任务）。



### 为什么你脚手架项目的部署方案要区分单机版和集群版？

在设计脚手架项目的部署方案时，区分单机版和集群版主要基于以下考虑：

- 单机版：适用于开发测试、小型项目或个人学习，资源需求低（如1核2G服务器）。快速启动，无需复杂依赖。
- 集群版：面向生产环境、高并发或企业级应用，需要高可用和横向扩展能力。依赖分布式组件部署。

提供两种方案有以下好处：

- 降低门槛：单机版让用户快速体验核心功能，避免初期陷入复杂部署。
- 灵活演进：用户可从单机版过渡到集群版，逐步适应生产需求。

### 为什么将房源表拆分为水平分表（如按状态分表）？具体如何实现？

- 原因：
  - 单表数据量过大（如房源状态频繁更新）会导致查询性能下降；
  - 不同状态（上架/已租/下架）的查询频率不同，拆分可减少锁争用；
  - 便于维护（如清理历史数据时不影响活跃数据）。
- 实现：
  - 通过`house_status`表独立存储状态字段，业务层通过`house_id`关联查询。更新状态时只需操作小表，核心SQL示例：
  - ```SQL
    -- 查询已出租房源
    SELECT h.* FROM house h 
    JOIN house_status s ON h.id = s.house_id 
    WHERE s.status = 'RENTING';
    ```



### 描述房源缓存的“双写一致性”方案，如何解决缓存穿透？

- 双写一致性：采用 Cache-Aside 模式：
  - 写 DB 后同步更新 Redis（加分布式锁保证原子性）；
  - 读请求先查缓存，未命中则查DB并回填。
- 缓存穿透解决：
  - 空值缓存：查询DB不存在时，在Redis缓存`NULL`（设置 TTL）；
  - 在Redis层预过滤非法`house_id`。



### 为什么用CompletableFuture获取城市列表？对比传统串行调用的优势？

### 策略模式如何解决筛选条件的扩展问题？新增一个“是否地铁房”筛选条件需改动哪些代码？

- 新增`SubwayFilter`实现`IHouseFilter`
- 在请求DTO中添加字段`Boolean requireSubway`

```TypeScript
@Componentpublic 
class SubwayFilter implements IHouseFilter {
    @Override
    public Boolean filter(HouseDTO house, SearchHouseListReqDTO req) {
        // 从房源标签判断是否近地铁
        return req.getRequireSubway() == null 
            || house.getTags().contains("near_subway");
    }
}
```

无需修改主流程，新增类自动被`houseFilterMap`收集

### 为什么用Redisson而不是Redis SETNX实现分布式锁？

SETNX 缺陷：

- 需自行处理锁续期（业务执行时间长导致锁过期）；
- 释放锁时需Lua脚本保证原子性（避免误删其他线程锁）。

Redisson 优势：

- Watchdog 机制：后台线程自动续期（默认30s）；
- 可重入锁：支持同一线程多次加锁；
- API封装：`lock()`/`unlock()`方法开箱即用。

### 咨询消息的持久化方案如何优化？

分库分表：按`user_id`哈希分库，按月份分表（如`chat_msg_2023_10`）；

读写分离：

- 写操作：主库（高并发下队列缓冲）；
- 读操作：从库 + Redis缓存最近10条消息；

冷热分离：超过3个月的消息转存至Elasticsearch。
